70 /*
71  JOIN_CACHE is the base class to support the implementations of both
72  Blocked-Based Nested Loops (BNL) Join Algorithm and Batched Key Access (BKA)
73  Join Algorithm. The first algorithm is supported by the derived class
74  JOIN_CACHE_BNL, while the second algorithm is supported by the derived
75  class JOIN_CACHE_BKA.
76  These two algorithms have a lot in common. Both algorithms first
77  accumulate the records of the left join operand in a join buffer and
78  then search for matching rows of the second operand for all accumulated
79  records.
80  For the first algorithm this strategy saves on logical I/O operations:
81  the entire set of records from the join buffer requires only one look-through
82  the records provided by the second operand.
83  For the second algorithm the accumulation of records allows to optimize
84  fetching rows of the second operand from disk for some engines (MyISAM,
85  InnoDB), or to minimize the number of round-trips between the Server and
86  the engine nodes (NDB Cluster).
87 */

598 /*
599  The class JOIN_CACHE_BKA_UNIQUE supports the variant of the BKA join algorithm
600  that submits only distinct keys to the MRR interface. The records in the join
601  buffer of a cache of this class that have the same access key are linked into
602  a chain attached to a key entry structure that either itself contains the key
603  value, or, in the case when the keys are embedded, refers to its occurance in
604  one of the records from the chain.
605  To build the chains with the same keys a hash table is employed. It is placed
606  at the very end of the join buffer. The array of hash entries is allocated
607  first at the very bottom of the join buffer, then go key entries. A hash entry
608  contains a header of the list of the key entries with the same hash value.
609  Each key entry is a structure of the following type:
610  struct st_join_cache_key_entry {
611  union {
612  uchar[] value;
613  cache_ref *value_ref; // offset from the beginning of the buffer
614  } hash_table_key;
615  key_ref next_key; // offset backward from the beginning of hash table
616  cache_ref *last_rec // offset from the beginning of the buffer
617  }
618  The references linking the records in a chain are always placed at the very
619  beginning of the record info stored in the join buffer. The records are
620  linked in a circular list. A new record is always added to the end of this
621  list. When a key is passed to the MRR interface it can be passed either with
622  an association link containing a reference to the header of the record chain
623  attached to the corresponding key entry in the hash table, or without any
624  association link. When the next record is returned by a call to the MRR
625  function multi_range_read_next without any association (because if was not
626  passed together with the key) then the key value is extracted from the
627  returned record and searched for it in the hash table. If there is any records
628  with such key the chain of them will be yielded as the result of this search.
629 
630  The following picture represents a typical layout for the info stored in the
631  join buffer of a join cache object of the JOIN_CACHE_BKA_UNIQUE class.
632 
633  buff
634  V
635  +----------------------------------------------------------------------------+
636  | |[*]record_1_1| |
637  | ^ | |
638  | | +--------------------------------------------------+ |
639  | | |[*]record_2_1| | |
640  | | ^ | V |
641  | | | +------------------+ |[*]record_1_2| |
642  | | +--------------------+-+ | |
643  |+--+ +---------------------+ | | +-------------+ |
644  || | | V | | |
645  |||[*]record_3_1| |[*]record_1_3| |[*]record_2_2| | |
646  ||^ ^ ^ | |
647  ||+----------+ | | | |
648  ||^ | |<---------------------------+-------------------+ |
649  |++ | | ... mrr | buffer ... ... | | |
650  | | | | |
651  | +-----+--------+ | +-----|-------+ |
652  | V | | | V | | |
653  ||key_3|[/]|[*]| | | |key_2|[/]|[*]| | |
654  | +-+---|-----------------------+ | |
655  | V | | | | |
656  | |key_1|[*]|[*]| | | ... |[*]| ... |[*]| ... | |
657  +----------------------------------------------------------------------------+
658  ^ ^ ^
659  | i-th entry j-th entry
660  hash table
661 
662  i-th hash entry:
663  circular record chain for key_1:
664  record_1_1
665  record_1_2
666  record_1_3 (points to record_1_1)
667  circular record chain for key_3:
668  record_3_1 (points to itself)
669 
670  j-th hash entry:
671  circular record chain for key_2:
672  record_2_1
673  record_2_2 (points to record_2_1)
674 
675 */
